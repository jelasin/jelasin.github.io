沙箱 Pwn

前置知识

沙箱保护是对程序加入一些保护，是程序运行过程中的一种隔离机制，其目的是限制不可信进程和不可信代码的访问权限。seccomp是内核中的一种安全机制，seccomp 规则的设置，可以使用 prctl 函数和 seccomp 函数族。最常见的是禁用一些系统调用，如execve，使得我们不能通过系统调用 execve 或 system 等获取到远程终端权限，因此只能通过ROP的方式调用 open, read, write的来读取并打印 flag 内容(open系统调用实际上是调用了openat，所以直接 调用openat，然后除了 read，write，其实还有两个readv，和writev，这些就能绕过限制读取flag,有些连openat都禁用的，可以 ptrace 修改syscall)。可以利用 seccomp-tools 来查看是否开启了沙箱，以及沙箱中一些允许的 syscall。一般有两种函数调用方式实现沙盒机制，第一种是采用 prctl 函数调用，第二种是使用seccomp 库函数。

prctl

函数原型：

prctl 是基本的进程管理函数，最原始的沙箱规则就是通过 prctl 函数来实现的，它可以决定有哪些系统调用函数可以被调用，哪些系统调用函数不能被调用。

  • PR_SET_SECCOMP(22)：当第一个参数是 PR_SET_SECCOMP, 第二个参数 argv2 为 1 的时候，表示允许的系统调用有 read，write，exit 和 sigereturn；当 argv 等于2的时候，表示允许的系统调用由argv3指向sock_fprog结构体定义，该结构体成员指向的sock_filter可以定义过滤任意系统调用和系统调用参数。
  • PR_SET_NO_NEWPRIVS(38)，第二个参数设置为1，表示禁用系统调用 execve() 函数，同时，这个选项可以通过 fork() 函数和clone() 函数继承给子进程。

seccomp

prctl 函数是最原始的，建立沙箱规则的函数。后面出现的seccomp.h对prctl函数进行了封装，有了seccomp_init()，seccomp_rule_add()，seccomp_load()函数来进行沙箱规则的设置。

  • __int64 seccomp_init(uint32_t def_action)：
  • int seccomp_rule_add(scmp_filter_ctx ctx, uint32_t action, int syscall, unsigned int arg_cnt, ...);
  • int seccomp_load(scmp_filter_ctx ctx);

seccomp_load()负责导入规则，seccomp_load()导入规则之后，添加的规则才能执行。

setcontext+orw

glibc < 2.29

glibc-2.27

Glibc-2.27 的 ORW 解题思路主要是劫持 free_hook 或者 malloc_hook写入 setcontext函数中的 gadget，free 时 rdi 为堆地址，通过 rdi索引，来设置相关寄存器，并执行提前布置好的 ORW ROP chains，我们通常从 setcontext + 53 开始给各个寄存器赋值。mov rcx, [rdi+0A8h] 和 push rcx 实质上是在给我们的 rip 进行赋值（最后 ret --> pop rip 将会把 rcx 的值赋给 rip），再次 ret 将会调用我们 rsp 所指向的 ORW 链。

申请堆块前，先看一下 seccomp 影响的堆风水。

将 tcache_key 破坏后再利用 tcache double free 可以泄露出 heap 信息。我们需要借助 tcache_0x80 来申请到 tcache_perthread_struct 结构体，并将 tcache_perthread_struct.counts[0x23] 标记的数量改为 \x07，下次释放 0x250 大小的 tcache_perthread_struct 将会被放进 unsorted bin 中。由此泄露出 libc 信息。

  • 构造 ORW 链。

图示

我们控制了 tcache_perthread_struct 结构体也就控制了整个 tcache，可以用来任意地址写，我们把free_hook指向 setcontext+53，调用 free 函数从而执行我们的控制链。接下来详细讲解以下调用流程。我们调用 free 函数时，rdi 指针指向了 heap_base+0x2000 : stack_addr_1，并且会开始执行 setcontext+53，此时 rdi+0xa0 的位置为 stack_addr_2，成功把 rsp指向stack_addr_2，而 stack_addr_2 保存了我们的 orw 调用链，并且 mov rcx, [rdi+0xa8]; push rcx 成功将 ret 指令地址压入栈中，最后调用原有的 setcontext+127 处的 ret==pop rip 指令时会把 rip 指向我们通过 rcx 压入栈中的 ret 指令，再次执行 ret 指令时会调用到 stack_addr_2 保存的 orw 调用链。

2.31 > glibc >= 2.29

glibc-2.29

在 glibc-2.29 以后，索引变成了 rdx。在glibc-2.29到glibc-2.30 的偏移为 setcontext+53。

glibc >= 2.31

glibc-2.31

到了 glibc-2.31 以后我们控制的偏移变成了 setcontext+61。

高版本沙箱

可用手法：

  • gadget_1：

这个 gadget 可以在 glibc 2.29 到 glibc 2.32 之间使用。通过 rdi 来控制 rdx 后可以通过 setcontext 继续控制其他寄存器和执行流。

  • gadget_2：

这个 gadget 可以通过 rdi 控制 rbp 的值，进而控制 rax 完成跳转。如果我们给 rax 传递一个 leave; ret == (mov rbp, rsp; pop rbp; pop rip;) 地址，便可以完成栈迁移。

  • 通过 __environ 泄露栈地址

泄露栈地址后可以通过一些堆利用手法申请到栈上的地址，将返回地址处改为我们的 ORW。
